# 3 **Estratégias de Engenharia de Software**

## 3.1 **Estratégia Priorizada**

**Abordagem de Desenvolvimento de Software:** Ágil 

**Ciclo de Vida**: Iterativo e incremental

**Processo de Engenharia de Software:**: Kanban/XP

  1. #### **Quadro Comparativo**

|          Critério           | Cascata                                                                                                   | Kanban/XP                                                                                                                                                          |
| :-------------------------: | --------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
|      **Flexibilidade**      | Baixa, todos os requisitos devem ser elicitados no começo do projeto.                                     | Alta, garante que os desenvolvedores possam trabalhar na construção do produto requisito a requisito, evitando sobrecarga inicial indesejada e abraçando mudanças. |
|        **Entregas**         | A entrega é feita no final do projeto completo.                                                           | A entrega é feita de maneira incremental ao longo do projeto, com deploys intermediários.                                                                          |
| **Envolvimento do Cliente** | Pontual, mais ao início do projeto para construção de requisitos e regras de negócio.                     | Contínuo, ao longo do projeto serão feitas reuniões para construir em conjunto os requisitos.                                                                      |
|  **Adaptação a Mudanças**   | Difícil, pois a falta de flexibilidade por conta da elicitação de requisitos feitas no início do projeto. | Fácil, permite construção contínua de requisitos e especificações ao longo do projeto.                                                                             |
|   **Validação Contínua**    | Em períodos maiores, porém com pouco envolvimento do cliente a não ser no início.                         | Em períodos menores, como de 15 em 15 dias, para verificar se requisitos estão sendo feitos corretamente, assim como validar assim que estiverem feitos.           |

## 3.2 **Justificativa**

O uso do **Kanban/XP** permite interações constantes com o cliente, entregas rápidas e feedbacks frequentes, o que é ideal para este projeto que será desenvolvido e validado diretamente com o dono da academia.
